Based on the logs you provided, I can see exactly what is happening:

1.  **MySQL is Healthy**: The MySQL logs show it initialized correctly and is "ready for connections".
2.  **Backend is FAILING**: The backend logs show a `java.sql.SQLException: Access denied for user 'jobportal'@'172.18.0.4'`.

### **The Cause**
Even though we deleted the bad script, the **MySQL user `'jobportal'` was already created** in the previous initialization (when the bad script existed) or by the Docker image with a password that **doesn't match** what the backend is trying to use.

The error `Access denied` confirms the user exists but the password is wrong.

### **The Solution: Hard Reset**
We need to **completely destroy the database volume** one last time to force MySQL to re-create the user with the *correct* password from your `.env` file.

I know we tried this, but it seems the volume wasn't actually wiped cleanly or the bad script was still lingering in the volume's init directory.

### **Plan**
1.  **Stop everything.**
2.  **Prune everything:** I will give you a command to aggressively remove the volume.
3.  **Verify .env:** We will double-check that the `.env` file on the server actually has the password you expect.
4.  **Restart.**

### **Commands for You**
Run these exact commands on your server:

1.  **Stop and Remove Volume:**
    ```bash
    cd /opt/jobportal
    docker-compose -f docker-compose.prod.yml down -v
    docker volume prune -f
    ```
    *(The `-v` flag is crucial—it deletes the volumes associated with the containers)*

2.  **Verify .env Password (Optional but recommended):**
    ```bash
    cat .env | grep MYSQL_PASSWORD
    ```
    *(Make sure this matches what you think it is)*

3.  **Start Fresh:**
    ```bash
    docker-compose -f docker-compose.prod.yml up -d
    ```

4.  **Check Logs:**
    ```bash
    docker logs jobportal-backend-prod --tail 50
    ```

This **MUST** work because it forces a 100% clean start.
